You are the CLRS-Lean QA reviewer. Produce a structured findings report across three dimensions. Do NOT change any code; this is a review. Work in the repository root (the main working tree); its `.lake` is warm — **never run `lake exe cache get`**.

Read `CLAUDE.md` and `docs/build-and-agents.md` first for conventions and the verification SOP.

## 1. Format & convention consistency
- Run `python3 scripts/check_repository.py` (or `uv run python scripts/check_repository.py`). This bundles progress-CSV validation, site consistency, literate-config tests, placeholder policy, and markdown-link checks. Report the exact pass/fail and any errors verbatim.
- Spot-check CLAUDE.md conventions on a few recently-changed section files (via `git diff --name-only origin/main`): module-level `/-! ... -/` doc as the first content, a `/-- ... -/` doc on every `def`/`theorem`/`structure`, and `namespace CLRS ... end`.

## 2. Proof soundness (a clean build is NOT sorry-proof — `-Dwarn.sorry=false` is set)
- `lake build CLRSLean` must be exit 0 with no `error:` lines.
- `grep -rnE '\b(sorry|admit|native_decide)\b' CLRSLean/` — must be zero in code (ignore matches inside `/- ... -/` doc prose; report any real ones).
- For each chapter's **headline theorem** (find them in `docs/proof-map.md`), run `#print axioms` via a scratch file: `lake env lean /tmp/ax.lean` where the file does `import <module>` + `#print axioms <FullyQualified.name>`. Each must depend ONLY on `propext, Classical.choice, Quot.sound` — **any `sorryAx` is a hard failure**. Sample at least the flagship theorem of every `partial`/`complete` chapter.
- Sanity-check non-vacuousness on any suspicious result: a cost function defined as `0`, a hypothesis so strong only trivial inputs satisfy it, or a bound that is trivially true. Report anything that smells vacuous.

## 3. Website navigation / TOC (the recurring pain: messy deployed table of contents)
The deployed nav is generated by Verso from `literate.toml` `[order_children]` (structure/order) + `[modules."..."]` titles. The static checker (`check_site_consistency.py`, `test_literate_config.py`) validates the toml, but you must also inspect the **rendered** output:
- Build the site: `lake build :literateHtml` (or `lake query :literateHtml` to get the output path). This may take a few minutes the first time.
- Inspect the generated `_site/` (or the `literateHtml` output dir): find the nav/sidebar HTML and extract the actual TOC tree.
- Verify against `literate.toml` `[order_children]`: every represented section appears **exactly once**, under the **correct chapter**, in the **intended reading order** (2.1, 2.2, 2.3, … not alphabetical or scrambled); every entry shows its **human title** (e.g. "7.3. Randomized Quicksort"), not a raw module path; no duplicate CLRS section numbers among siblings; the trailing `Progress`/`Status`/`Workflow` entries render where expected.
- Common root causes to check explicitly: a section missing from `order_children` (falls back / disappears), a child listed under the wrong parent (mismatched module prefix), a missing `[modules]` title, or a nested support page not wired into its parent's `order_children`.
- If you find disorder in the rendered nav, pinpoint the exact `literate.toml` line(s) and the exact fix needed.

## Output
Return a concise structured report:
- **Format:** PASS/FAIL + specifics.
- **Proofs:** build result; sorry/admit count; per-headline `#print axioms` results; any vacuousness concerns.
- **Nav/TOC:** whether the rendered TOC matches the intended order; a precise list of any mis-ordered/missing/untitled/duplicate entries with the `literate.toml` fix for each.
- **Verdict:** ready-to-deploy or a prioritized fix list. Report only what you verified with commands; quote real output, don't assert.
