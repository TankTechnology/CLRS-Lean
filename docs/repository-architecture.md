# Repository Architecture

This document defines the stable layers of CLRS-Lean and the ownership rules
between them.  It is the maintainer-facing companion to the shorter repository
layout in the top-level README.

## Design Principles

1. Formal statements and proofs live in Lean, not in status prose.
2. Chapter-level progress has one machine-readable source.
3. Reader navigation and maintainer planning are separate concerns.
4. Shared abstractions are introduced only after multiple chapters need them.
5. Mathematical correctness may be sealed before RAM or pointer-level
   refinement, provided the boundary is explicit.

## Layer Model

```text
section proof modules
        |
        v
chapter guide/aggregator modules
        |
        +----> interface and closure tests
        |
        v
CLRSLean.lean library root and Verso landing page
        |
        v
literate.toml navigation -> Verso HTML -> GitHub Pages

section/chapter changes
        |
        +----> docs/clrs-proof-progress.csv -> CLRSLean/Progress.lean
        +----> docs/proof-map.md
        +----> docs/proof-status-board.md when priorities change
```

## Lean Library Layer

### Library Root

`CLRSLean.lean` imports every public chapter guide, the reusable proof-pattern
aggregator, and the public progress/status/workflow pages.  It should remain a
short landing page, not a second theorem ledger.

### Chapter Guides

`CLRSLean/Chapter_XX.lean` imports the represented sections for one chapter and
states the chapter's formalization boundary.  A guide answers:

- Which sections are represented?
- What is proved for the current model?
- Which representation or cost refinements remain?

It should not restate every helper theorem.

### Section Modules

`CLRSLean/Chapter_XX/Section_XX_Y_Name.lean` owns definitions, executable
models, theorem interfaces, and proofs.  Section modules are the authority for
what is formally true.  Helper files with the same section number are allowed
when a proof stack has a real dependency boundary, as in Chapter 22 DFS.

### Reusable Proof Patterns

`CLRSLean/ProofPatterns/` contains small abstractions shared by more than one
chapter.  These modules capture proof geometry such as boundary shifts,
exchange certificates, fibers, and interval relations.  They must not absorb
chapter-specific algorithms merely to reduce local line counts.

### Tests

`Tests/` contains stable interface and closure checks.  Interface tests verify
that downstream users can consume the advertised declarations without relying
on implementation details.  Closure tests seal a milestone against accidental
reintroduction of placeholders or weakened theorem boundaries.

## Progress And Status Layer

The status documents have deliberately different jobs:

| Artifact | Ownership |
| --- | --- |
| `docs/clrs-proof-progress.csv` | Chapter rows, counts, status labels, and remaining core groups |
| `CLRSLean/Progress.lean` | Generated public rendering of the CSV |
| `docs/proof-map.md` | Detailed theorem names, proof patterns, and exact gaps |
| `docs/proof-status-board.md` | Compact planning buckets and next priorities |
| `CLRSLean/Status.lean` | Concise reader-facing interpretation of the current status |
| `docs/status/blocked-and-deferred.md` | Design blockers and intentionally deferred layers |
| `docs/proof-audits/` | Dated evidence for a particular closure or review |

`CLRSLean/Progress.lean` is generated by
`scripts/check_progress_csv.py --write-dashboard`.  The generated file must not
be edited independently.

## Website Layer

`literate.toml` owns navigation order and page titles.  The Verso build starts
from `CLRSLean.lean`, renders imported literate modules, applies the repository
CSS, and writes generated HTML under `.lake/build/`.  The Pages workflow then
checks, optimizes, and deploys that generated output.

Generated HTML and `_site/` are artifacts, not source files.

## Tooling Layer

- `scripts/check_repository.py` runs the fast repository-wide metadata and
  configuration checks.
- `scripts/check_progress_csv.py` validates the progress schema and regenerates
  the public dashboard.
- `scripts/check_site_consistency.py` checks Lean modules, navigation, chapter
  guides, and the source catalog.
- `scripts/test_literate_config.py` verifies navigation order against imports.
- rendering and optimizer scripts validate the generated website.
- `uv.lock` and `pyproject.toml` define the Python helper environment.

Local Python commands should use `uv run python ...`.  CI may invoke the same
dependency-free scripts with `python3` after checkout.

## Dependency Rules

1. A section may import earlier sections or a reusable proof-pattern module.
2. A chapter guide imports its sections; sections do not import the guide.
3. The library root imports chapter guides, not individual section modules.
4. Progress and status prose must not be imported by proof modules.
5. Tests may import public chapter modules or focused section modules.
6. A sealed chapter may be extended only through a clearly named refinement
   layer or a new theorem target, not through silent weakening of its closure
   interface.
7. A cross-chapter implementation adapter belongs to the consuming chapter.
   For example, the union-find/Kruskal adapter is nested under Section 23.2 and
   imports Chapter 21; Chapter 21 does not import the MST development.

## Change Matrix

| Change | Files that normally move together |
| --- | --- |
| New helper lemma | Section module; focused test when public |
| New public theorem | Section, chapter guide, proof map, progress CSV, dashboard |
| New section | Section, chapter guide, `literate.toml`, docs source catalog, progress CSV |
| Status transition | Progress CSV, generated dashboard, proof map, status board if priorities change |
| New shared abstraction | `ProofPatterns/`, aggregator, navigation/title config, dependent sections |
| Website-only change | Lean prose page or CSS, `literate.toml`, rendering checks |
| Sealed milestone | Interface test, closure test, dated audit, progress/status update |

## Verification Gates

Use three gates:

1. **Static gate:** `uv run python scripts/check_repository.py`.
2. **Lean gate:** current file, immediate dependents, then `lake build CLRSLean`.
3. **Site gate:** `lake build :literateHtml` plus rendering/freshness checks.

The static gate establishes metadata consistency.  It does not replace the Lean
kernel build, and the Lean build does not replace rendered-site checks.
